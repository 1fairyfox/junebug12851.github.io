## Synthesized themes

### Theme 1 — The `--ff-only` refresh fails on every hub force-push (RECURRING, highest priority)

Both nodes hit it independently. Each tried the documented `git pull --ff-only` to refresh
its read-only hub clone; both aborted with `fatal: Not possible to fast-forward` because the
hub's `dev` had been force-pushed (history rewritten). They recovered differently:

- pokered-save-editor-2 → delete + fresh re-clone (the runbook's stated fallback).
- random-ai-prompt → the lighter `git fetch` + `git reset --hard origin/dev` on the mirror only.

Both explicitly ask that the fallback stop being a parenthetical and become a first-class
step printed right next to the `--ff-only` line. random-ai-prompt notes the `fetch` +
`reset --hard origin/dev` form is lighter than a full re-clone and works because the mirror
is disposable and git-ignored. This is the strongest signal in the set: two of two nodes,
same failure, same request.

- **Touches:** `hub/standards/adopting-updates.md` (refresh step) and `hub/standards/cross-project-sync.md` (the sync/refresh description). The node-side mirrors of cross-project-sync should follow once the canonical text changes.
- **Proposed change:** Promote the fallback to a numbered step beside `--ff-only`: "If the pull aborts (hub `dev` is force-pushed routinely), refresh the reference clone with `git fetch origin dev` + `git reset --hard origin/dev` — this is safe because the clone is a disposable, git-ignored mirror — or delete and re-clone shallow. Never `reset --hard` anything but the mirror."

### Theme 2 — "What changed since last adoption" is hard to compute (RECURRING)

pokered-save-editor-2 states it directly; random-ai-prompt touches the same nerve from the
lucky side. The node keeps no record of which hub VERSION it last adopted, and because hub
`dev` is force-pushed, the previously-cloned commit can vanish — so `git log old..new` may be
impossible and the delta has to be reconstructed from commit subjects + version numbers
("guess-adjacent," in P2's words). random-ai-prompt got an exact old→new diff only because
both commits happened to still be retained that run — i.e. it worked by luck, not by design.
Both also note that diffing a canonical hub file against a heavily path-adapted node copy is
noisy and not a reliable "what changed upstream" signal.

- **Touches:** `hub/standards/adopting-updates.md` ("See what changed" step). Optionally a small node-side marker convention.
- **Proposed change:** (a) Lead the "see what changed" guidance with "skim the hub's `notes/version/` changelog as the primary signal" rather than file diffs, and call out that canonical-vs-adapted diffs are noise. (b) Give nodes a lightweight "last adopted hub version" marker (a line in the node's `cross-project-sync.md`, or a field under `notes/fairyfox-reports/`) so the next adoption has a bounded "since X" that survives a force-push.

### Theme 3 — CI-owns-tagging conflicts with the manual-tag release step (ONE-OFF, high impact)

pokered-save-editor-2 only. Its `release.yml` already derives `v<VERSION>` and creates the
tag itself on the `main` push, and its `version` job skips the release if the tag already
exists. Following the runbook literally — hand-pushing `git tag -a vX.Y.Z && git push --tags`
— would make every release silently no-op. It had to diverge: the pipeline owns tagging, the
release merge to `main` carries no hand tag. One node, but it would bite any node with a
tag-gated `release.yml`, so worth a standing warning.

- **Touches:** `hub/standards/git-workflow.md` ("Cutting a release") and `hub/standards/adopting-updates.md` (release step).
- **Proposed change:** Add: "If the project's release pipeline creates the version tag, do NOT also tag by hand — the merge to `main` is the release act and CI applies the tag; a hand-pushed tag will make a tag-gated release run skip itself."

### Theme 4 — Report ambiguity when a check converts straight into an adopt (ONE-OFF)

random-ai-prompt only. The standard wants a report per run, but this run was a check that the
owner immediately green-lit into an adopt in the same session. RAP wrote one combined
`adopting-updates` report rather than splitting check + adopt; the runbook doesn't say which
to do.

- **Touches:** `hub/standards/process-reports.md`.
- **Proposed change:** State that when a check converts into an adopt in the same session, one combined `adopting-updates` report covers both — no separate check report needed.

### Theme 5 — Chicken-and-egg: check-only reporting on a not-yet-adopted node (ONE-OFF, logical gap)

random-ai-prompt only, but it's a genuine contradiction in the standard. The new rule says even
a check-only run writes a report into `notes/fairyfox-reports/` — but on a node that hasn't
adopted process-reports yet, that folder doesn't exist, and creating it is itself an act of
adoption, which the check-only flow forbids.

- **Touches:** `hub/standards/process-reports.md` (and the check-only path in `adopting-updates.md`).
- **Proposed change:** Resolve explicitly — either a check-only run on a not-yet-adopted node reports inline and defers the written report until adoption, or folder-creation is permitted even in check-only. Pick one and state it.

### Theme 6 — Volunteer a "what changed, where" close-out after an adopt (ONE-OFF, ties to Theme 1)

random-ai-prompt only. After the adopt, the owner asked "is my repository alright online and
locally? what did you do why?" The run had touched a `feature/` branch, merged to `dev`,
pushed `dev`, and `reset --hard` the git-ignored hub mirror — and that combination read as
alarming without a plain summary. The `reset --hard` especially needs loud framing that it hit
the reference clone, never project history.

- **Touches:** `hub/standards/adopting-updates.md` (end of runbook).
- **Proposed change:** End the adopting-updates runbook with a standard close-out the node volunteers unprompted: local vs `origin` for `dev`/`main` (identical?), `main` untouched, working tree clean, and an explicit "the only `reset --hard` was on the disposable mirror, not the project."
